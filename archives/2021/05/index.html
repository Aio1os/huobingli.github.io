Mysql 引擎

MyIASM
不支持事务
不支持外键
不支持 MVCC
表锁
select count(*) 不需要全表扫描
因为 MyIASM 保存了表的行数
索引采用 B+树
如果表的读操作远远多于写操作时，并且不需要事务的支持的。可以将MyIASM作为数据库引擎的首先。

InnoDB
支持事务
支持外键
支持 MVCC
默认行级锁
select count(*) 需要全表扫描
索引采用B+树

ACID特性

特性 描述 -
Atomicity（原子性） 要么都成功，要么都不成功 -
Consistency（一致性） 事务前后数据的完整性必须保持一致 -
Isolation（隔离性） 事务之间互不影响 -
Durability（持久性） 事务一旦被提交，对数据的修改就是持久有效的 -

Mysql事务隔离级别

https://zhuanlan.zhihu.com/p/43493165
级别 | 描述 | 加锁方式

  • | - | -
    读未提交 read_uncommitted | 会导致脏读、不可重复读、幻读 | -
    读已提交 read_committed| 会导致不可重复读、幻读 | 行锁
    可重复读 repeatable_read| 会导致幻读 | 检索条件有索引时：next key; 检索条件没有索引时：表锁
    串行化 Serializable| - | 表锁

mysql默认级别是 repeatable_read

隔离问题

隔离问题 描述 -
脏读 A 事务读取了 B 事务修改的数据，但是如果 B 事务回滚了，此时 A 事务读取的数据就成了脏数据 -
不可重复读 A 多次读取了某个数据，事务 B 修改了该数据会导致多次读取数据不一致 -
幻读 A 事务想要修改某些数据，假如修改的过程中事务 B 又新增了一条数据，A 事务会没有修改全，方法产生了幻读 -

按照锁的机制分

锁机制 作用
排他锁 写锁，其他事务不能读取，也不能写。
共享锁 读锁，其他事务可以读，但不能写。

按照锁的粒度分

- 开销 加锁效率 粒度 冲突概率 并发度
表锁 开销小 加锁快；不会出现死锁 锁定粒度大 发生锁冲突的概率最高 并发度最低
行锁 开销大 加锁慢，会出现死锁 锁定粒度最小 发生锁冲突的概率最低 并发度也最高
页锁 开销中 加锁中，会出现死锁 锁定粒度中等 - 并发度一般

MyISAM 和 MEMORY 存储引擎采用的是表级锁（table-level locking）
BDB 存储引擎采用的是页面锁（page-level locking），但也支持表级锁
InnoDB 存储引擎既支持行级锁（row-level locking），也支持表级锁，但默认情况下是采用行级锁。
